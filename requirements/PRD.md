# PRD: AMD 双卡 GPU 任务调度器

> 版本: v0.1-draft | 作者: amd-pm | 日期: 2026-05-18

---

## 1. 产品概述

### 1.1 产品名称
AMD 双卡 GPU 任务调度器（AMD Dual-GPU Task Scheduler）

### 1.2 核心价值
智能分配双 AMD GPU 任务，最大化双卡利用率，降低任务等待时间。通过高效的调度策略和自动化管理，让双卡计算资源发挥最大价值。

### 1.3 目标用户
- **深度学习开发者**：同时训练多个模型，需要合理分配 GPU 资源
- **GPU 计算用户**：科学计算、渲染、数据处理等需要 GPU 加速的场景
- **DevOps 工程师**：管理多用户 GPU 集群，需要任务排队和资源隔离

### 1.4 技术栈
Go 1.26.1 + ROCm/OpenCL + gRPC

## 2. 功能需求

### P0 — 必须实现（MVP）

#### F-001 任务提交与队列管理
- 支持通过 CLI/API 提交 GPU 计算任务
- 任务队列：FIFO + 优先级混合队列
- 任务状态跟踪：Pending → Running → Completed / Failed
- 支持任务取消和超时自动终止
- 任务元数据记录：提交时间、运行时长、GPU 占用、退出码

#### F-002 双卡负载均衡
- 实时感知两张 AMD GPU 的负载状态（显存占用、计算利用率）
- 根据负载情况将新任务分配到较空闲的 GPU
- 避免单卡过载导致 OOM 或性能瓶颈
- 支持配置每张 GPU 的最大并发任务数

### P1 — 重要功能

#### F-003 任务优先级调度
- 支持高/中/低三级优先级
- 高优先级任务可抢占低优先级任务的 GPU 资源（可配置）
- 紧急任务支持插队，跳过排队直接调度
- 优先级可通过 API 动态调整

#### F-004 GPU 状态监控
- 实时采集 GPU 硬件指标：利用率、显存、温度、功耗
- 通过 gRPC 流式推送 GPU 状态更新
- 提供简化的 CLI 仪表盘（`scheduler status`）
- 历史指标数据保留（可选持久化到本地文件）

### P2 — 增强功能

#### F-005 故障自动切换
- GPU 故障检测：连续超时、驱动错误、通信失败
- 自动将故障 GPU 上的任务迁移到健康 GPU
- 故障 GPU 自动隔离，恢复后自动重新纳入调度池
- 故障事件通知（日志 + 可选 webhook）

#### F-006 REST/gRPC API
- 提供完整的 RESTful HTTP API 供外部集成
- gRPC 双向流接口用于实时状态推送
- API 认证与鉴权（Token / mTLS）
- OpenAPI / Protobuf 文档自动生成

## 4. 调度策略说明

### 4.1 轮询调度（Round-Robin）

**原理**：按固定顺序依次将任务分配到 GPU 0 和 GPU 1。

**适用场景**：任务规模相近、显存需求均匀的场景。

```
Task 1 → GPU 0
Task 2 → GPU 1
Task 3 → GPU 0
Task 4 → GPU 1
...
```

**优点**：实现简单、分配均匀。
**缺点**：不考虑实际负载差异，可能导致负载不均。

### 4.2 最少连接优先（Least-Loaded）

**原理**：每次调度时选择当前任务数最少（或综合负载最低）的 GPU。

**判断指标**：
- 当前运行任务数
- 显存占用率
- GPU 计算利用率

**综合得分公式**（初始版本）：

```
score = α × (running_tasks / max_tasks) + β × (vram_used / vram_total) + γ × (gpu_util / 100)
```

其中 α, β, γ 为可配置权重（默认 α=0.4, β=0.4, γ=0.2）。

**适用场景**：任务规模差异大、显存需求不均匀的场景。

**优点**：动态适应负载变化，最大化资源利用率。
**缺点**：需要实时监控数据，调度开销略高。

### 4.3 资源预留（Reservation）

**原理**：允许为高优先级任务预留 GPU 资源，确保关键任务能在指定时间内获得 GPU。

**规则**：
- 可配置预留 GPU 数量（1 或 2）和预留时长
- 预留期间，该 GPU 只接受指定优先级及以上的任务
- 超过预留时长未使用，自动释放预留资源
- 低优先级任务在预留 GPU 上运行时，可被抢占

**适用场景**：生产环境中有 SLA 要求的关键训练任务。

**优点**：保证关键任务的及时执行。
**缺点**：预留期间可能降低整体 GPU 利用率。

### 4.4 策略选择建议

| 场景 | 推荐策略 |
|------|----------|
| 任务规模均匀、无优先级需求 | 轮询调度 |
| 任务规模差异大 | 最少连接优先 |
| 有 SLA 要求的关键任务 | 资源预留 |
| 通用生产环境（默认） | 最少连接优先 + 资源预留 |

## 5. 非功能性需求

| 维度 | 要求 |
|------|------|

| 可扩展性 | 设计支持未来扩展至 4 卡/8 卡 |
